## OpenCut Classic 改造技术方案与实施规划

### 一、项目现状分析

OpenCut Classic 是一个开源 Web 视频编辑器，核心技术栈为 Next.js 16 + React 19 + Rust/WASM GPU 合成引擎。项目采用 Monorepo 结构（Turborepo + Bun workspaces），前端使用 Zustand 管理状态、Radix UI 构建组件、Tailwind CSS 4 样式，后端仅用于认证和反馈，所有项目数据和素材均存储在浏览器本地（IndexedDB + OPFS）。

**现有核心能力：**

编辑器已具备完整的时间轴多轨编辑体系，支持视频、图片、文本、贴纸、图形、音频、特效七种轨道类型，具备 Undo/Redo 命令系统、关键帧动画、GPU 加速合成（17 种混合模式）、蒙版羽化、变速、音频波形可视化等能力。渲染管线基于 Rust 编译的 WASM 模块，支持 WebGPU（优先）和 WebGL 双路径。字幕方面已有 SRT/ASS 导入解析和 Whisper 语音转文字能力。导出支持 MP4/WebM 格式。

**现状与目标的差距：**

| 能力维度 | 现状 | 目标要求 | 差距评估 |
|---------|------|---------|---------|
| 素材存储 | 纯客户端（IndexedDB + OPFS） | 服务器端持久化，用户独立素材库 | **重大改造** |
| 用户体系 | Better Auth（独立注册） | 对接现有系统用户 | **需替换认证层** |
| 字幕编辑 | SRT/ASS 导入 + Whisper 转写 | 字幕编辑（含样式、位置、时间调整） | 中等增强 |
| 半透明水印 | 不存在 | 图片/文字水印，支持透明度调节 | **新增功能** |
| 标题编辑 | 基础文本元素（单一样式） | 丰富标题模板和样式 | 中等增强 |
| 视频素材上传 | 客户端本地处理 | 服务端上传、转码、存储 | **重大改造** |
| 界面语言 | 全英文硬编码 | 中文界面 | **需全面国际化** |
| 数字人口播集成 | 无 | 对接文稿生成数字人视频的流程 | **新增集成** |

### 二、目标架构设计

#### 2.1 整体架构

改造后的系统采用前后端分离架构，编辑器前端保留 GPU 渲染能力，所有数据和素材迁移至服务端：

#### 2.2 用户体系对接方案

现有系统使用 Better Auth 独立认证，需替换为与你的系统对接。推荐采用 **JWT Token 透传 + 用户信息同步** 模式：

```
你的系统用户登录 → 生成 JWT Token → 携带 Token 跳转编辑器
                                          │
                                    编辑器前端验证 Token
                                          │
                              ┌───────────┼───────────┐
                              │ 验证通过  │           │ 验证失败
                              ▼           │           ▼
                        提取用户信息      │     跳转登录页
                        创建/更新本地用户  │
                        进入编辑器        │
```

具体做法：后端 API 中间件拦截所有请求，从 Header 中提取 JWT Token 并验证签名，解析出 `userId`、`username`、`role` 等信息注入到请求上下文中。首次访问时自动在 PostgreSQL 中创建对应用户记录（懒注册），后续请求直接关联。这样既不影响编辑器独立运行，又能无缝对接你的用户体系。

#### 2.3 服务端素材存储架构

这是最核心的改造点。现有的 `StorageService`（IndexedDB + OPFS）需替换为服务端 API 驱动的存储层：

```
前端编辑器                          后端 API                        存储
    │                                  │                              │
    │  上传素材 (multipart/form-data)   │                              │
    ├─────────────────────────────────►│                              │
    │                                  │  1. 保存到对象存储            │
    │                                  ├─────────────────────────────►│
    │                                  │  2. 写入素材元数据到 PG       │
    │                                  ├─────────────────────────────►│
    │  3. 返回素材信息 (含 URL/ID)      │                              │
    │◄─────────────────────────────────┤                              │
    │                                  │                              │
    │  加载素材 (签名 URL 或流式传输)   │                              │
    ├─────────────────────────────────►│                              │
    │                                  │  生成临时签名 URL             │
    │                                  ├─────────────────────────────►│
    │  返回签名 URL                    │                              │
    │◄─────────────────────────────────┤                              │
    │                                  │                              │
    │  <video src="签名URL"> 直接播放   │                              │
```

**数据库设计（新增核心表）：**

```sql
-- 用户素材表
CREATE TABLE materials (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id),
    name        VARCHAR(255) NOT NULL,
    type        VARCHAR(20) NOT NULL,  -- 'video', 'image', 'audio', 'subtitle', 'watermark'
    mime_type   VARCHAR(100),
    file_size   BIGINT,
    storage_key VARCHAR(500) NOT NULL, -- 对象存储路径
    duration    FLOAT,                 -- 视频/音频时长（秒）
    width       INT,                   -- 视频/图片宽度
    height      INT,                   -- 视频/图片高度
    thumbnail_key VARCHAR(500),        -- 缩略图存储路径
    metadata    JSONB DEFAULT '{}',    -- 扩展元数据
    created_at  TIMESTAMPTZ DEFAULT now(),
    updated_at  TIMESTAMPTZ DEFAULT now()
);

-- 用户项目表
CREATE TABLE projects (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id),
    name        VARCHAR(255) NOT NULL,
    settings    JSONB DEFAULT '{}',    -- FPS、宽高比、背景色等
    scenes      JSONB DEFAULT '[]',    -- 场景与轨道数据（序列化）
    thumbnail_key VARCHAR(500),
    duration    FLOAT DEFAULT 0,
    created_at  TIMESTAMPTZ DEFAULT now(),
    updated_at  TIMESTAMPTZ DEFAULT now()
);

-- 导出记录表
CREATE TABLE exports (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id  UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    user_id     UUID NOT NULL REFERENCES users(id),
    format      VARCHAR(10) NOT NULL,  -- 'mp4', 'webm'
    quality     VARCHAR(20),
    storage_key VARCHAR(500) NOT NULL,
    file_size   BIGINT,
    status      VARCHAR(20) DEFAULT 'pending', -- 'pending','processing','done','failed'
    created_at  TIMESTAMPTZ DEFAULT now()
);
```

**索引策略：** 为 `materials.user_id`、`projects.user_id` 建立索引，支持按用户快速过滤；为 `materials.type` 建立复合索引，支持按类型筛选素材库。

### 三、功能模块详细设计

#### 3.1 字幕编辑模块

现有能力：SRT/ASS 文件解析（`src/subtitles/parse.ts`）、字幕到 TextElement 的转换（`src/subtitles/insert.ts`）、Whisper 语音转写。

**需增强的能力：**

字幕样式编辑器——在属性面板中为字幕 TextElement 增加专属编辑区，包括字体选择（预置中文常用字体：思源黑体、思源宋体、站酷系列等）、字号与颜色、描边与阴影、背景半透明条、位置预设（上/中/下，支持九宫格定位）。

字幕时间轴微调——支持在时间轴上拖拽调整单条字幕的起止时间，拖拽字幕边缘进行时长伸缩，批量调整字幕时间偏移（整体前移/后移）。

字幕批量编辑——支持字幕文本的批量修改（双击编辑）、字幕样式的全局应用（修改一条，全部同步）、字幕的导入/导出（SRT 格式）。

数字人口播字幕联动——从数字人系统导入文稿时，自动按句/段落分割生成字幕轨道，时间戳与数字人视频音频对齐。

#### 3.2 半透明水印模块

这是完全新增的功能。水印作为一种特殊的叠加层，需独立于普通轨道元素：

**水印类型：**

图片水印——上传 PNG/SVG Logo，支持透明度调节（0-100%）、位置预设（四角 + 居中）、缩放比例、平铺模式（单个/平铺/对角线）。

文字水印——自定义文字内容，支持字体、颜色、透明度、旋转角度、位置预设。

**实现方案：**

水印在数据模型上作为项目的 **全局属性** 而非轨道元素存储，这样它会自动叠加在所有场景的每一帧上，不会被用户误删或移动。渲染时，水印层作为合成管线的最后一个 Layer 注入到 `FrameDescriptor` 中，位于所有轨道元素之上。在编辑器预览画布中，水印始终可见但不可选中（避免干扰编辑），通过属性面板中的"水印设置"Tab 进行配置。

**数据存储结构：**

```typescript
interface WatermarkSettings {
  enabled: boolean;
  type: 'image' | 'text';
  // 图片水印
  imageMaterialId?: string;   // 关联素材库中的图片
  // 文字水印
  text?: string;
  fontFamily?: string;
  fontSize?: number;
  fontColor?: string;
  // 通用属性
  opacity: number;            // 0-1，半透明度
  position: 'top-left' | 'top-right' | 'bottom-left' | 'bottom-right' | 'center';
  offsetX: number;            // 像素偏移
  offsetY: number;
  rotation: number;           // 旋转角度
  scale: number;              // 缩放比例
  tileMode: 'none' | 'tile' | 'diagonal';  // 平铺模式
  tileGap?: { x: number; y: number };       // 平铺间距
}
```

#### 3.3 标题编辑模块

现有能力：基础 TextElement，支持字体、大小、颜色、对齐、背景等参数。

**需增强的能力：**

标题模板系统——预置 20-30 个中文标题模板（封面标题、章节标题、角标、下方条幅等），每个模板包含预设的字体组合、颜色方案、装饰元素、动画效果。标题模板作为"组合元素"插入，包含一个主文本层 + 可选的装饰图形层。

标题动画——预置入场/出场动画（淡入、滑入、缩放、打字机等），复用现有关键帧动画系统实现。

#### 3.4 视频素材上传与管理

**上传流程设计：**

```
用户选择文件 → 前端校验(类型/大小) → 分片上传到后端
                                          │
                                    后端接收分片并合并
                                          │
                              ┌───────────┼──────────────┐
                              ▼           ▼              ▼
                        保存到对象存储  异步转码缩略图  写入元数据到PG
                              │           │              │
                              ▼           ▼              ▼
                        返回素材ID和预览URL给前端
```

**分片上传：** 视频文件通常较大（几十MB到几GB），采用分片上传（每片 5MB）+ 断点续传机制，前端使用 `File.slice()` 分片，后端接收后合并。上传进度实时展示。

**服务端转码：** 上传完成后，触发异步转码任务（基于 FFmpeg 或云服务），生成低分辨率预览视频（用于编辑器预览，减少浏览器解码负担）和缩略图。

**素材库 UI：** 左侧素材面板改造为云端素材库视图，支持文件夹分类（视频、图片、音频、字幕）、搜索过滤、排序、批量选择、删除、重命名，拖拽到时间轴插入。

#### 3.5 中文界面国际化

当前项目无 i18n 框架，所有文本硬编码在组件中。推荐采用 **next-intl** 方案（与 Next.js App Router 原生兼容）：

**实施步骤：**

第一步，安装 next-intl 并配置中文 locale。在 `next.config.ts` 中注册 i18n 插件，创建 `messages/zh-CN.json` 翻译文件。

第二步，系统性提取所有硬编码英文字符串。涉及范围包括编辑器面板标签（Media→素材、Text→文字、Stickers→贴纸、Sounds→音效、Effects→特效、Captions→字幕、Settings→设置等）、属性面板字段名、工具栏按钮文本、上下文菜单项、对话框文本、Toast 通知消息、错误提示信息。

第三步，将提取的字符串写入翻译文件，组件中使用 `useTranslations()` hook 替换硬编码文本。

第四步，调整字体配置，将默认字体从 Inter 改为中文友好字体栈：`"Inter", "PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif`。

**预估翻译条目：** 基于项目组件数量和内容复杂度，预计需要翻译 300-500 个字符串条目。

### 四、与数字人口播系统的集成方案

你的系统已能根据文稿生成数字人口播视频，集成方案如下：

```
┌─────────────────┐         ┌──────────────────┐         ┌─────────────────┐
│  文稿编辑系统    │         │  数字人渲染引擎   │         │  视频编辑器     │
│  (你的系统)     │         │  (你的系统)       │         │  (OpenCut改造)  │
└────────┬────────┘         └────────┬─────────┘         └────────┬────────┘
         │                           │                             │
         │ 1. 用户编写文稿            │                             │
         │ ─────────────────────────►│                             │
         │                           │ 2. 生成口播视频              │
         │                           │ ───────────────────────────►│
         │                           │   (视频文件 + 时间轴元数据)  │
         │                           │                             │
         │                           │ 3. 文稿自动拆分为字幕        │
         │                           │ ───────────────────────────►│
         │                           │   (字幕文本 + 时间戳)        │
         │                           │                             │
         │ 4. 用户在编辑器中          │                             │
         │    调整字幕、添加水印、     │                             │
         │    添加标题、混剪素材       │                             │
         │                           │                             │
         │ 5. 导出最终视频            │                             │
         │◄─────────────────────────────────────────────────────────│
```

**集成接口设计：**

接口一：`POST /api/projects/from-digital-human`，接收数字人视频 URL + 文稿字幕数据，自动创建项目并导入视频和字幕轨道。

接口二：`GET /api/materials/sync`，从数字人系统同步用户已有的视频素材到编辑器素材库。

接口三：`POST /api/exports/callback`，导出完成后回调通知你的系统，更新文稿关联的最终视频。

### 五、技术栈选型建议

| 领域 | 推荐方案 | 理由 |
|------|---------|------|
| 后端 API | Next.js API Routes + tRPC 或 Hono | 复用现有 Next.js 项目，减少跨端通信复杂度 |
| 数据库 | PostgreSQL 17（已有 docker-compose） | 现有基础设施，Drizzle ORM 已集成 |
| 缓存/队列 | Redis 7（已有） | 上传任务队列、签名 URL 缓存 |
| 对象存储 | MinIO（自建）或 阿里云 OSS | MinIO 兼容 S3 协议，可本地开发；OSS 适合生产 |
| 文件上传 | 分片上传 + tus 协议 | 断点续传、进度追踪 |
| 视频转码 | FFmpeg + Bull/BullMQ 任务队列 | 异步处理，不阻塞主线程 |
| 认证对接 | JWT 验证中间件 | 无状态、简单、与你系统对接成本低 |
| 国际化 | next-intl | Next.js App Router 原生支持，TypeScript 友好 |
| 部署 | Docker Compose（开发）/ K8s（生产） | 现有已有 docker-compose.yml 基础 |

### 六、实施路线图（调整版：先独立可演示，后改造对接）

核心原则：**先用最快速度做出可独立演示的中文编辑器**，再逐步改造素材库，最后开发对接接口。所有改动按"是否可以纯前端完成"来决定优先级。

```

纯前端改动                      后端API + 存储                 认证 + 集成接口
保留 IndexedDB 存储             逐步替换存储层                 接口开发放最后
                               
✅ 中文界面                     ✅ 素材上传API                ✅ JWT认证对接
✅ 半透明水印                   ✅ 对象存储(MinIO/OSS)        ✅ 数字人系统导入API
✅ 字幕编辑增强                 ✅ 云端素材库UI               ✅ 素材同步API
✅ 标题模板系统                 ✅ 项目云端保存               ✅ 导出回调API
✅ 数字人文稿导入字幕(前端)     ✅ 视频转码缩略图             ✅ 端到端联调
```

本阶段目标：产出一个中文界面、具备水印/字幕/标题能力的完整视频编辑器，可以直接部署演示。所有数据继续使用现有 IndexedDB + OPFS 存储，不触碰后端。

这两项完全独立，可以并行开发。

| 位置 | 文件 | 字符串数 | 关键翻译 |

*字幕编辑增强（约 3 天）*

现有能力（SRT/ASS 解析、Whisper 转写、字幕插入）完全可用，重点增强编辑体验：

1. **字幕样式编辑** — 在属性面板中，当选中字幕类型的 TextElement 时，增加"字幕样式"区域。实现方式：在 `panels/properties/registry.tsx` 的 `getPropertiesConfig()` 中，检测元素是否来自字幕轨道（通过 `element.name` 前缀或新增一个 `isSubtitle` 标记），若是则在现有 Text Tab 基础上追加字幕专属控件：描边（strokeWidth + strokeColor）、阴影（shadowX/Y/blur/color）、背景半透明条（backgroundEnabled + backgroundColor + backgroundOpacity）。预置中文常用字体列表：思源黑体、思源宋体、阿里巴巴普惠体、站酷快乐体等（通过 Google Fonts 或直接打包 woff2）。

3. **字幕样式全局同步** — 新增"应用到所有字幕"按钮，点击后将当前字幕的样式参数（字体、字号、颜色、描边、背景）批量写入同一轨道中所有字幕 TextElement。实现方式：遍历字幕轨道的所有元素，通过 `UpdateElementsCommand` 批量更新，作为一个原子操作支持 Undo。

   - 数字标注（圆形底 + 数字）

2. **模板 UI** — 在左侧素材面板的 Text Tab 中（`text/components/assets-view.tsx`），将现有的单一"Default text"扩展为模板网格。每个模板显示缩略图预览（用 Canvas 离屏渲染生成），点击或拖拽插入时间轴。插入时调用模板工厂函数生成元素，通过 `InsertElementCommand` 添加到轨道。

#### Phase 2：素材库云端化（约 3 周）——后端 API + 存储改造

本阶段目标：将素材上传、项目保存从浏览器本地迁移到服务端，实现用户独立的云端素材库。

**第 3-4 周：后端 API + 对象存储**

1. **数据库表结构** — 在 `src/db/schema.ts` 中新增 `materials`、`projects`（服务端版本）、`exports` 三张表（表结构见第二节 2.3）。使用已有的 Drizzle ORM + PostgreSQL。

2. **素材上传 API** — `POST /api/materials/upload`，支持分片上传（每片 5MB），使用 `File.slice()` 前端分片 + 后端合并。上传完成后异步生成缩略图（使用 sharp 处理图片，ffmpeg 截取视频首帧）。素材元数据写入 PostgreSQL，文件存储到 MinIO 或阿里 OSS。

3. **素材 CRUD API** — `GET /api/materials`（列表，按 user_id 过滤）、`DELETE /api/materials/:id`、`GET /api/materials/:id/url`（生成临时签名 URL）。

4. **对象存储部署** — 在 `docker-compose.yml` 中添加 MinIO 服务，配置 bucket 和访问策略。生产环境切换为阿里 OSS。

5. **前端素材面板改造** — 将 `views/assets.tsx` 的上传流程从本地 IndexedDB 改为调用服务端 API。素材列表从服务端拉取，预览图使用签名 URL。保留 IndexedDB 作为本地缓存层（已下载的缩略图不重复请求）。

**第 5 周：项目云端保存 + 素材库 UI 完善**

1. **项目 CRUD API** — `POST /api/projects`、`GET /api/projects`、`PUT /api/projects/:id`、`GET /api/projects/:id`。项目的场景/轨道数据序列化为 JSONB 存储。

2. **存储层适配** — 在 `services/storage/service.ts` 上层增加 `RemoteStorageAdapter`，将 API 调用包装成与现有 `IndexedDBAdapter` / `OPFSAdapter` 兼容的接口。`SaveManager` 的自动保存从写 IndexedDB 改为调 API（加 debounce，避免过于频繁）。

3. **素材库 UI** — 素材面板增加文件夹分类（视频、图片、音频、字幕）、搜索过滤、右键菜单（重命名、移动、删除）、批量操作。

4. **视频转码** — 上传后触发 FFmpeg 异步任务，生成 720p 预览版本。编辑器预览使用低分辨率版本（减少浏览器解码负担），导出时使用原始素材。

**Phase 2 交付物：** 编辑器具备云端素材库，用户上传的素材保存在服务端，项目自动保存到服务端，不同浏览器/设备可以访问同一份数据。

#### Phase 3：系统对接（约 2 周）——认证 + 集成接口

本阶段目标：将编辑器接入你的现有系统，实现用户打通和数字人视频工作流闭环。接口开发安排在此阶段最后。

**第 6 周：认证对接**

1. **JWT 中间件** — 新建 `src/middleware/auth.ts`，拦截所有 `/api/*` 请求，从 `Authorization: Bearer <token>` 中提取 JWT，验证签名后注入 `userId`、`username` 到请求上下文。

2. **移除 Better Auth** — 删除 `src/auth/` 目录和相关的 `/api/auth/[...all]` 路由。前端登录态改为从 URL 参数或 cookie 中读取 JWT。

3. **用户懒注册** — 首次请求时，根据 JWT 中的用户信息自动在 PostgreSQL 中创建用户记录，后续请求直接关联。

**第 7 周：集成接口开发**

这是最后的开发内容，放在最末尾是因为依赖前面所有基础设施就绪：

1. **`POST /api/projects/from-digital-human`** — 接收数字人视频文件 URL + 文稿字幕 JSON（含每句文本和时间戳），自动创建项目、下载视频到素材库、生成字幕轨道，返回项目 ID。你的系统生成数字人视频后调用此接口，然后将用户重定向到 `/editor/{projectId}`。

2. **`GET /api/materials/sync`** — 从数字人系统拉取用户已有的视频素材列表，同步到编辑器素材库（去重、增量同步）。

3. **`POST /api/exports/callback`** — 导出完成后，编辑器调用此接口通知你的系统，携带导出视频的 URL 和关联的文稿 ID，你的系统更新文稿记录中的"最终视频"字段。

4. **端到端联调** — 完整走通：文稿 → 数字人视频 → 编辑器打开 → 编辑字幕/水印/标题 → 导出 → 回调通知。

**Phase 3 交付物：** 编辑器完全融入你的系统，用户无感知地从数字人生成页进入编辑器，编辑后导出，视频自动回写。

### 八、关键风险与应对策略

**风险一：中文翻译量大导致 UI 错位。** 180 条字符串翻译后，中文宽度与英文差异可能导致按钮溢出、标签截断。应对：使用弹性布局（flex + `min-width`），对关键区域（导出弹窗、属性面板 Tab 栏）做逐像素验证。优先翻译核心编辑器界面，非核心页面保持英文。

**风险二：水印渲染精度。** 文字水印在 GPU 合成中需要先光栅化为纹理。应对：复用现有 TextNode 的 Canvas 2D 光栅化能力（`text/layout.ts` + `text/measure-element.ts`），将文字渲染到 OffscreenCanvas 后通过 `uploadTexture()` 上传为 GPU 纹理。平铺模式通过生成多个 Layer 实现，每层共享同一纹理。

**风险三：存储迁移复杂度高（Phase 2）。** 现有 `StorageService` 有 31 个版本的迁移逻辑，与 IndexedDB/OPFS 深度耦合。应对：不直接替换，而是在 `StorageService` 上层增加 `RemoteStorageAdapter`，将 API 调用包装成与现有 Adapter 兼容的接口。IndexedDB 降级为本地缓存，加速重复加载。

**风险四：WASM 渲染与大文件预览性能（Phase 2）。** 浏览器解码高分辨率视频消耗大量内存和 CPU。应对：服务端上传后自动生成 720p 预览版本，编辑器使用预览版编辑，导出时服务端用原始素材合成。
